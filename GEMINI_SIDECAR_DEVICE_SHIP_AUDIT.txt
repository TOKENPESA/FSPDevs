GOAL
----
Ship Fiber Agent Sidecar as a separate-device app (Windows desktop and/or Android)
that can:
  1) Install/run on a retail/agent laptop or phone
  2) Register itself to the live MFA droplet (mfa.fsprotocol.com / 167.99.150.153)
  3) Use Fiber Network Node (FNN) for real testnet payments (not silent demo)
  4) Operate DiCoBa / mobile-money modules as a kiosk / agent console

VERDICT (as of this snapshot)
-----------------------------
Control plane (MFA TLS defaults + auto-register + Tauri shell + operator UI) is
substantially implemented.

Product shipping as a SELF-CONTAINED Windows/Android app that BUNDLES live FNN
and settles Fiber payments end-to-end is NOT ready.

FNN on the laptop is an EXTERNAL process today (fnn-testnet/fnn.exe → :8227).
If FNN is down, Sidecar silently falls back to SimulatedFnnClient while
FNN_MODE may still say "testnet" — operators can confuse demo receipts with live
payments.

Android: env prep scripts exist; `tauri android init` / gen/android project does
NOT exist in-repo yet. On-device FNN is not packaged.

CRITICAL GAPS — WINDOWS APP ON SEPARATE DEVICE
----------------------------------------------
1. FNN NOT BUNDLED
   tauri.conf.json has no externalBin / sidecar fnn.exe.
   Edge defaults to http://127.0.0.1:8227. Without local FNN → simulate.

2. HUB FNN RPC NOT FOR EDGE CUSTODY
   Treasury RPC is loopback / nginx-restricted to MFA. Retail Windows FA cannot
   safely use Hub :8227 as its own funding wallet RPC.

3. SILENT SIMULATE FALLBACK UX RISK
   UI can appear "testnet" while DiCoBa contributions are local-only when FNN
   is down or FNN_MODE=simulate. Need hard "Demo vs Live" gating for ship.

4. STATE DIR / INSTALLER
   Identity/state often under relative .fiber-agent/ — fragile for installed apps.
   Need absolute FIBER_AGENT_STATE_DIR + proper Windows installer (tauri build /
   NSIS) and first-run story.

5. RELEASE SIGNING KEYS
   Telemetry / OOB signing needs per-device FSP_AGENT_SECRET_KEY (or allowed
   production secret path). Dev-key short-circuits are not ship-safe.

6. CHANNEL / LIQUIDITY WIZARD MISSING
   fund CKB → open Fiber channel → ready is still script/ops driven
   (fnn-cli, setup-public-channel.ps1, droplet scripts). Not an in-app wizard.

7. DiCoBa P2P SETTLEMENT
   Intended: FA → vault leader_pubkey via FNN send_payment + local receipt.
   Local/simulate path skips mesh settlement. Leader pubkey in UI is still
   often a placeholder (e.g. 03leader_test).


CRITICAL GAPS — ANDROID APP
---------------------------
1. tauri android init NOT DONE — no src-tauri/gen/android
2. SDK/NDK/JAVA_HOME + rust Android targets required; no CI green APK proof
3. Cleartext/network security for any local HTTP FNN not designed (TLS MFA OK)
4. JoyID / passkey return path in WebView unproven (portal already opens external)
5. On-device Fiber/FNN binary packaging DOES NOT EXIST
   Mobile with FNN_MODE=testnet will almost always simulate unless architecture
   chooses: companion PC FNN, remote gated FNN API, or a new light-client design
6. Tauri capabilities still desktop-oriented (windows: main*)

REMAINING MILESTONES (recommended)
----------------------------------

M1  Decide FNN product story: (A) bundle/side-install local fnn on Windows,
    (B) official "demo edge / Hub custody", or (C) remote FNN API — stop implying
    bundled FNN if not shipping it
M2  Hard Demo vs Live UX + disable silent fake "testnet" when simulate backend
M3  Identity + mesh-pubkey registry for each edge device
M4  Liquidity: fund CKB + open ≥1 Fiber channel; one live DiCoBa send_payment
M5  Windows installer (tauri build), icons, absolute STATE_DIR, first-run guide
M6  Android scaffold: tauri android init, NDK CI, cleartext policy, capabilities
M7  Android networking architecture decision (no on-device FNN in current repo)
M8  In-app fund → channel → contribute wizard; harden register rate limits
